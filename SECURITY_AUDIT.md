# Security audit — MoTrend

**Дата:** 2025-03-01  
**Область:** Firestore rules, Auth flow, Storage upload, Cloud Functions triggers.

---

## Критичные проблемы

### 1. Пользователь может выставить себе любые кредиты (`users`)

**Где:** `firestore.rules` — коллекция `users`.

**Суть:** Правило `allow read, write: if request.auth != null && request.auth.uid == userId` разрешает полный доступ на запись к своему документу. Клиент может вызвать `updateDoc(users/{uid}, { creditsBalance: 999999 })` и получить произвольное число кредитов.

**Риск:** Обход монетизации, бесплатное использование платных лимитов.

**Рекомендация:** Запретить клиенту менять `creditsBalance`. В правилах разрешать только чтение этого поля и запись остальных полей профиля (email, language, country, updatedAt). Либо списывать/начислять кредиты только из Cloud Functions, а в правилах требовать `request.resource.data.creditsBalance == resource.data.creditsBalance` при update.

---

### 2. Клиент может подделать результат джобы (`jobs`)

**Где:** `firestore.rules` — коллекция `jobs`.

**Суть:** Для своих джоб разрешён `update` без ограничения полей. Пользователь может вызвать:

```js
updateDoc(doc(db, "jobs", jobId), {
  status: "done",
  outputVideoUrl: "https://evil.com/fake.mp4",
  updatedAt: serverTimestamp()
});
```

**Риск:** Подмена результата в UI, возможное распространение ссылок на вредоносный контент как «результат генерации».

**Рекомендация:** Разрешить клиенту при update менять только поля, которые он реально выставляет после загрузки фото: `inputImageUrl`, `inputImagePath`, `updatedAt`. Остальные поля (status, outputVideoUrl, errorMessage и т.д.) — только из Cloud Functions (через Admin SDK).

---

## Средние риски

### 3. Нет ограничения типа файла в Storage

**Где:** `storage.rules` — `user_uploads/{uid}/{allPaths=**}`.

**Суть:** Правила проверяют только `request.auth.uid == uid`. Тип контента (image/jpeg и т.д.) и расширение не проверяются. В путь всегда пишется `photo.jpg`, но клиент может отправить любой blob с любым content-type.

**Риск:** Загрузка исполняемых файлов под видом «фото» в свой каталог; при дальнейшей интеграции (например, CDN/превью) возможны XSS или обход проверок.

**Рекомендация:** В Cloud Function после загрузки (или в триггере по Storage) проверять content-type и, при необходимости, метаданные/магические байты. В Storage rules при необходимости можно добавить проверку `request.resource.contentType.matches('image/.*')` для `user_uploads`.

---

### 4. Нет rate limiting на createJob

**Где:** `functions/src/index.ts` — `createJob`.

**Суть:** Любой аутентифицированный пользователь может вызывать `createJob` неограниченно. Каждый вызов создаёт документ в Firestore и возвращает путь для загрузки.

**Риск:** DoS, рост затрат на Firestore/Storage и последующие сервисы (Kling и т.д.).

**Рекомендация:** Ввести лимит по пользователю (например, N джоб в час/день) через Firestore (счётчик/агрегат) или через отдельный сервис; при превышении возвращать `resource-exhausted`.

---

### 5. CORS: true на createJob

**Где:** `functions/src/index.ts` — `onCall({ cors: true }, ...)`.

**Суть:** Вызов доступен с любого origin.

**Риск:** Сайт с другого домена может встроить вызов и при наличии токена пользователя создавать джобы от его имени (CSRF при условии украденного/утеченного токена). Для callable-функций Firebase проверяет только токен, не origin.

**Рекомендация:** Если приложение должно быть доступно только с вашего домена, ограничить CORS в настройках функции или на уровне Firebase Hosting/прокси. Для многих SPA это средний риск, если токены хранятся аккуратно.

---

## Что сделано хорошо

- **Auth:** Вход только через Firebase Auth (email/password и Google). Нет своих паролей, нет хранения секретов на клиенте.
- **Jobs create:** Создавать джобы может только Cloud Function; клиент не может создать документ в `jobs` напрямую (create: if false).
- **Storage path:** Путь загрузки выдаётся сервером (`user_uploads/{uid}/{jobId}/photo.jpg`), подмена чужого `uid` в пути отсекается правилами Storage.
- **Триггер по джобе:** Срабатывает по факту появления `inputImageUrl` и статуса `queued`; не доверяет произвольным полям от клиента для логики обработки (кроме того, что клиент может перезаписать status/output — см. п. 2).
- **Шаблоны:** Только чтение, запись закрыта — ок.
- **Профиль users:** Ограничение «только свой документ» по uid — ок, но нужно ограничить поля (п. 1).

---

## Рекомендуемые правки правил (кратко)

1. **users:** При `update` требовать `request.resource.data.creditsBalance == resource.data.creditsBalance`. При `create` допускать только если `creditsBalance` не передан или равен 0.
2. **jobs:** При `update` разрешать изменение только ключей `inputImageUrl`, `inputImagePath`, `updatedAt` (например, через `affectedKeys().hasOnly([...])`).
3. **Storage (опционально):** Для `user_uploads` добавить проверку `request.resource.contentType.matches('image/.*')`.

После правок стоит прогнать тесты и вручную проверить сценарии: логин → создание джобы → загрузка фото → обновление джобы → проверка, что кредиты и статус джобы клиент изменить не может.
